## Artifact One: Algorithms and Data Structures - Python-based DQN Project, CS350

[View Artifact One](https://github.com/brianac02/SNHU-CS350)

The presented artifact is a Python-based Treasure Hunt application originally developed in my CS350 course, Current and Emerging Trends in Computer Science, taken in the first few months of 2026. The program uses deep Q-learning to train a pirate agent to navigate an 8x8 maze, avoid blocked spaces, and reach a treasure. Over time, the model improves its decision making by learning from experience through replay memory and iterative training. As an artifact, it represents both my exposure to emerging areas of computer science and my ability to apply those concepts in a functional software solution.

I selected this artifact for my ePortfolio because it demonstrates both technical depth and growth as a software developer. The original project already showed my ability to work with reinforcement learning concepts, neural network design, and algorithmic problem solving. However, the enhanced version better reflects the level of professionalism expected in a capstone because it moves beyond simply functioning code and toward a more maintainable, usable, and polished application. It stands out amongst many of the other project that I have worked on in the sense that it displays a rudimentary understanding of the way AI works, an area of computer science that is growing all in the sense of availability, complexity, functionality, funding, and popularity. Thus, I deemed it important to include an understanding of the subject matter as it will likely prove immediately useful in my professional life. 

Several parts of the artifact highlight my software development skills. The training workflow demonstrates my ability to design and evaluate a computing solution built on algorithmic principles, while the neural network, replay memory, and training loop show my understanding of modern computing tools and techniques. Through my enhancement work, I improved the structure of the project by refactoring large blocks of code into helper functions, replacing hardcoded values with configurable inputs, adding validation and error handling, and implementing safeguards to prevent runaway training episodes. I also improved the presentation of the artifact by organizing training output more clearly and building a more interactive notebook interface so the project feels closer to an application than a simple console program. These enhancements allowed me to demonstrate several important skills: refactoring for readability and maintainability, defensive programming, debugging, interface design, and the ability to apply emerging tools in a way that produces a stronger end product. More importantly, the enhancement process shows that I can revisit an existing artifact, evaluate its weaknesses, and improve it in ways that make it more professional and more valuable to a user.

The enhancement process taught me that strong software development involves much more than making a program work once. A project can be technically correct and still need significant improvement in organization, usability, maintainability, and clarity. As I worked through this artifact, I learned the importance of separating core logic from configuration, validating inputs before execution, and structuring code so that it can be more easily understood and extended later. I also gained a stronger appreciation for the balance between experimentation and stability, especially in a project involving reinforcement learning, where relatively small changes can have large and sometimes unpredictable effects on performance.

One of the biggest challenges I faced was improving the artifact without overcomplicating it. During the enhancement process, I had to evaluate which changes genuinely improved the project and which changes introduced unnecessary complexity. I also had to work through issues related to training consistency, runtime, and making the notebook feel more polished without breaking the working version of the program. Feedback played an important role in that process. As I revised the artifact, I used feedback to simplify areas that had become too experimental, clarify the training workflow, improve the readability of the code, and shift the project toward a cleaner, more user-focused presentation.

Overall, the artifact was improved in several meaningful ways. It became more modular, easier to configure, more resilient to invalid input, and more accessible to users through clearer output and a more polished interface. In terms of course outcomes, I believe this artifact most strongly demonstrates my ability to design and evaluate computing solutions using appropriate computer science practices, as well as my ability to apply well-founded and innovative tools to implement solutions that deliver value. I also believe it supports the outcome focused on developing a security mindset, particularly through the use of validation, safeguards, and defensive programming practices that reduce the likelihood of misuse or failure. In addition, the written and visual improvements made to the project support the communication outcome by showing that I can present technical work in a more professional and understandable way. The collaborative-environment outcome was addressed less directly in this particular artifact, since this project was primarily an individual technical implementation rather than one centered on team-based decision making. However, the effort to make the artifact clearer, more organized, and more adaptable for users does still reflect an awareness of designing technical work for broader audiences.

## Artifact Two: Software Design and Engineering - C++ OpenGL Application, CS330

[View Artifact Two](https://github.com/brianac02/SNHU-CS330)

This artifact was originally created as the final project for my CS330: Computational Graphics and Visualization course, which I completed in Fall 2025. The project is a C++ OpenGL application that renders a 3D nature scene using textured objects, lighting, and user-controlled camera movement. The program is structured around a main file along with supporting SceneManager and ViewManager classes. SceneManager is responsible for preparing and rendering the objects in the scene, while ViewManager handles the camera, projection, and user input. I chose this artifact for my ePortfolio because it demonstrates both technical depth and growth in my software development skills. It shows that I can work with graphics programming, object-oriented design, user interaction, and rendering logic in a more advanced environment than a basic console application.

I included this artifact because it highlights several important strengths in software development, especially in structure, problem-solving, and maintainability. The original project already demonstrated my ability to build a working graphics application with multiple interacting components, including texture loading, material setup, transformations, lighting, and camera controls. However, the enhancement process gave me the opportunity to improve the project in a more professional way. I refactored repetitive rendering logic in RenderScene() into a reusable helper function, replaced many hard-coded values with named constants, reduced reliance on global state by moving camera and interaction-related state into ViewManager, and improved error handling for initialization, texture loading, and material lookup. These changes made the project cleaner, easier to read, easier to maintain, and more reliable. Through this enhancement, I demonstrated skills in refactoring, modular design, defensive programming, debugging, and improving code readability. One thing it doesn’t necessarily show is artistic ability – the meshes and textures presented aren’t advanced or professional. The reason being is that the purpose behind the project is knowing how to do it, not how well it can be implemented, and therefore the somewhat lackluster scene can be excused by the professionality of the code itself. 

Enhancing this artifact taught me a lot about the difference between writing code that simply works and writing code that is maintainable and scalable. One of the biggest things I learned was how quickly repetitive code and global state can make a project harder to understand and extend. Breaking rendering behavior into reusable functions made the code much easier to follow, and replacing hard-coded values with named constants made the scene configuration more readable and less error-prone. Another important lesson which was more reinforced rather than learned was the value of defensive programming. As I improved the project, I realized that several functions assumed everything would always work correctly, especially during setup and resource loading. Strengthening those areas made the application more professional and helped me think more seriously about reliability and failure cases.

A major challenge during the enhancement process was making structural improvements without breaking the project. Because the scene depended on multiple files, libraries, and project settings, even small refactors could create compile or linker errors if the header declarations, source definitions, or file paths did not stay consistent. I also had to troubleshoot issues caused by outdated project paths and dependency references, which reminded me that software development is not just about writing logic, but also about understanding how the full project environment fits together. I incorporated feedback from earlier analysis, code review observations, and testing throughout the enhancement process. That feedback helped me focus not just on visual output, but on the overall quality of the code, especially in areas like modularity, error handling, and maintainability.

The artifact was significantly improved through these enhancements. The rendering logic is now more modular, the configuration of scene objects is clearer, the camera and callback behavior are better encapsulated, and the project handles several types of failures more safely than before. In terms of course outcomes, this artifact most strongly supports designing and evaluating computing solutions, using well-founded techniques and tools to implement computer solutions, and developing a security-minded and defensive approach to software design. It also partially supports the communication outcome through the way I documented, analyzed, and justified the enhancement process in my ePortfolio. The outcome I feel is least directly represented in the artifact itself is the one related to building collaborative environments for diverse audiences, since this was primarily an individual technical project. Even so, presenting the artifact in a professional ePortfolio and explaining the reasoning behind the enhancements helps connect it to that broader outcome as well.

## Artifact Three: Databases - Java-based Weight Tracker App for Android, CS360

[View Artifact Three](https://github.com/brianac02/SNHU-CS360)

This third artifact is a mobile Weight Tracker application originally created as the final project for CS 360, Mobile Architect and Programming, which I completed from January to March of 2026. The application was designed to allow users to create an account, log in, set a target weight, record daily weight entries, and review past progress through a clean mobile interface. The project uses an Android-based architecture with a local database to store user information, goal weights, and daily weight entries. At its core, the application was intended to provide a simple and practical tool for tracking weight-loss progress while also demonstrating foundational mobile development skills such as user interface design, data storage, input handling, and user interaction.

I selected this artifact for inclusion in my ePortfolio because it provided a strong opportunity to demonstrate growth across software design and engineering, databases, and security-minded development within a single project. Similar to my reasoning for selecting the machine learning application, this is the only project that displays my ability to work within a mobile application development setting. The original version of the application was functional and already included several important mobile programming components, such as account creation, login functionality, a dashboard screen, daily weight entry, and persistent storage through SQLite. These components showcase my ability to build a multi-screen mobile application, organize application logic across activities and helper classes, connect front-end user interactions to back-end data storage, and implement CRUD-related database operations in a practical setting.

What made this artifact especially valuable for my ePortfolio was its potential for meaningful enhancement. The original version worked, but it also had several limitations that gave me a chance to demonstrate more advanced development skills. Through enhancement, I improved the application by redesigning the database structure so that weight entries and goal weights were tied to specific user accounts rather than being shared globally. I also improved session handling so users could remain signed in until they explicitly chose to sign out, which made the application more realistic and user friendly. In addition, I strengthened input validation and exception handling to reduce crashes caused by invalid or empty numeric input. I improved maintainability by moving hard-coded interface strings into string resources, and I improved password security by replacing weaker password-handling logic with a more secure salted hashing approach.

These enhancements significantly improved the artifact by making it more complete, more scalable, more secure, and more professional overall. They also allowed me to demonstrate specific software development skills including database redesign, relational thinking, secure coding practices, defensive programming, session management, mobile UI refinement, debugging, and testing. Rather than simply building a working app, I was able to show that I can revisit an existing codebase, identify weaknesses, and improve it in ways that better reflect real software engineering practices.
Enhancing this artifact taught me that software improvement is not just about adding new features, but about making an application more reliable, more maintainable, and more aligned with how it should actually function in a real-world setting. One of the biggest lessons I learned was how much impact database design has on the quality of an application. In the original version, the application functioned, but the data relationships were incomplete because weights and target goals were not truly tied to specific user accounts. Correcting that forced me to think more carefully about schema design, data ownership, and how one change in the database structure affects multiple parts of the application logic.

One of the most significant challenges I faced was implementing and verifying the multi-user system correctly. It was not enough to simply add a user ID field to a table; I also had to make sure that inserts, queries, session handling, and screen refresh behavior all respected the correct active user. This required more testing and troubleshooting than I originally expected. I also faced challenges related to session persistence and sign-out behavior, since I wanted the app to feel more natural to use without forcing a login every time it opened, while still allowing account switching within the app. Working through those issues helped me better understand how application state and user experience are closely connected.

Even moreso than the previous entries, defensive programming was a focus in the enhancement process. While the application seemed to work under ideal conditions, enhancing it made me focus more on edge cases, such as invalid numeric input, empty fields, database failures, and improper session state. By addressing those issues, I improved not just the app’s correctness, but also its stability and overall professionalism. In addition, I became more aware of the importance of secure data handling. Strengthening password storage and removing weaker fallback behavior helped me think more seriously about the role security should play even in smaller academic applications. The prior two artifacts didn’t handle user data such as usernames, emails, phone numbers, or passwords, so while input validation and such was important, having a secure password storage system, for example, didn’t need to be taken into consideration or taken as seriously, contrary to this program. The final enhanced version better separates user data, handles sessions more effectively, validates input more carefully, stores credentials more securely, and presents a cleaner, more maintainable codebase.

Through these enhancements, I feel that I most strongly met the course outcomes related to designing and evaluating computing solutions, using well-founded tools and techniques to implement valuable computer solutions, and developing a security mindset. More specifically, this artifact fully demonstrates my ability to design and improve a computing solution to better solve a defined problem, to apply practical development tools and techniques in a meaningful way, and to identify weaknesses in software design related to privacy, validation, and security. I also partially met the outcome related to professional communication, since presenting the artifact through the ePortfolio, code review, and narrative required me to explain technical decisions clearly to a broader audience. The course outcome related to building collaborative environments was the least directly represented in this artifact, since this project was completed individually rather than in a team-based environment. However, I still addressed that outcome to some extent by designing the application and its documentation in a way that would be understandable to different audiences and useful for communicating development decisions.
Overall, this artifact represents my growth from simply creating a functional mobile application to thinking more like a software developer who considers structure, maintainability, security, usability, and long-term improvement as essential parts of the development process.
